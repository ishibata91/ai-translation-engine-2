# Spec Gap Checklist

## 目的
実装不備なのか、仕様が古いのか、仕様が曖昧なのかを短時間で切り分けるための補助チェックリスト。

## 確認順
1. `docs/` に対象機能の canonical spec があるか。
2. 期待挙動が spec に明記されているか。
3. 複数 spec で同じ対象の記述が食い違っていないか。
4. 実装が spec から外れているのか、spec 自体が現実要件からズレているのか。
5. ユーザー確認なしでは決められない曖昧さが残っていないか。

## 実装不備と判断しやすい兆候
- spec の期待挙動が明確
- 実挙動がその期待から外れている
- 関連 spec 同士に矛盾がない
- ユーザー報告が spec と整合している

## 仕様が古いと判断しやすい兆候
- 現場の運用や他機能の現実挙動と spec がズレている
- docs が古い前提のまま残っている
- 変更済み UI や workflow に対して spec が追随していない

## 仕様が曖昧と判断しやすい兆候
- 期待結果が docs に書かれていない
- 成功時、失敗時、再試行時の扱いが抜けている
- UI の観測事実と内部処理の責務が混ざっている

## 仕様同士の矛盾と判断しやすい兆候
- 2 つ以上の spec が同じイベントに別の期待を書いている
- zone をまたいだ文書で責務境界が崩れている
- UI spec と workflow spec の主張が一致しない

## ユーザー確認が必要な条件
- spec だけでは正しい期待挙動を決められない
- 仕様同士が矛盾していて優先順位を決められない
- 修正案が既存 UX や運用フローを変えてしまう

## DB メンテナンスへ切り分ける兆候
- DB の整合性崩れや古いデータ残骸が主因で、アプリコードの通常フロー自体は contract どおり動いている
- 対処がデータ再生成、再投入、不要レコード削除、マイグレーション再実行で完結する
- code fix の提案が、異常データを見つけたときの自動補正や起動時 self-heal に偏っている
- 恒久修正より先に DB メンテ手順の明文化で再発を防げる
