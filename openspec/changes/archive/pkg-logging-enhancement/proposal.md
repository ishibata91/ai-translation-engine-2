# Proposal: pkg 全体へのログガイド第1部適用

## 1. 背景・目的
[`specs/log-guide.md`](specs/log-guide.md) の第1部に従い、AI解析に最適化された構造化ログ（JSON形式）を `pkg` パッケージ全体に導入する。
これにより、AIやエンジニアがシステムの問題を容易に追跡し、障害原因の特定やパフォーマンスのボトルネック発見を迅速に行えるようにする。

## 2. 変更内容の概要
- **外部システム境界 (I/O) のログ強化**: データベース操作、外部API通信、非同期通信の開始・終了時に詳細なパラメータと結果を記録する。
- **重要な条件分岐の決定理由の記録**: ビジネスロジックにおける重要な分岐点において、なぜそのルートを選択したかの理由（変数の値など）を記録する。
- **状態変化 (State Change) の記録**: エンティティのステータス遷移や重要なデータ更新の前後の状態を記録する。
- **異常系・バリデーションエラーのログ充実**: エラー発生時の不正な入力値やスタックトレースを [`telemetry.ErrorAttrs`](pkg/infrastructure/telemetry/span.go) を使用して統一的に記録する。
- **パフォーマンス計測の導入**: [`telemetry.StartSpan`](pkg/infrastructure/telemetry/span.go) を活用し、主要な処理ブロックの実行時間を記録する。

## 3. スコープ
- `pkg/` 配下の全パッケージ（`config`, `dictionary`, `export`, `infrastructure`, `parser`, `persona`, `pipeline`, `summary`, `task`, `terminology`, `translator`）
- `log-guide.md` 第1部の要件に基づいた実装を行う。

## 4. 期待される効果
- トレース可能性の向上: `trace_id` と `span_id` を介した処理の横断的追跡が可能になる。
- デバッグ効率の改善: AIがログを解析することで、プログラムのバグや設定ミスを自動的に指摘しやすくなる。
- パフォーマンス監視: どのコンポーネントで遅延が発生しているかを定量的に把握できる。

## 5. 検討事項・リスク
- **ログの肥大化**: すべての処理にログを仕込むとノイズが増えるため、`log-guide.md` の第2部「ログ挿入の判断フロー」に基づき適切に間引く。
- **パフォーマンスへの影響**: ログ出力自体がオーバーヘッドにならないよう、必要最低限の情報に絞り、構造化ログの生成コストに注意する。
- **機密情報**: パスワードや個人情報がログに出力されないよう、マスキングや除外を徹底する。
