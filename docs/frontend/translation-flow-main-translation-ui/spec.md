# 翻訳フロー 本文翻訳 UI

翻訳フローの `本文翻訳` タブにおける表示責務、状態遷移、ユーザー操作を定義する。

## Requirements

### Requirement: Translation Flow UI は `ペルソナ生成` の次に `本文翻訳` phase を表示しなければならない
システムは、`TranslationFlow` の UI において `本文翻訳` を `ペルソナ生成` の次、`export` の前に表示しなければならない。ユーザーは同じ translation project task の流れの中で本文翻訳 phase を開き、既存 task 再表示時も同じ phase として復元できなければならない。

#### Scenario: タブ順序に本文翻訳 phase が含まれる
- **WHEN** ユーザーが翻訳フロー画面を開く
- **THEN** UI は `データロード` `単語翻訳` `ペルソナ生成` `本文翻訳` `export` の順で phase を表示しなければならない
- **AND** `本文翻訳` は独立したタブ内容を持たなければならない

### Requirement: 本文翻訳タブは phase 専用のモデル設定と prompt を分離して表示しなければならない
システムは、本文翻訳 phase で使う翻訳モデル設定と user prompt を `translation_flow.translation` namespace に保存し、再表示時に復元しなければならない。system prompt は category / recordType ごとの同梱テンプレートから表示し、編集 UI を出してはならない。

#### Scenario: phase 専用設定が復元される
- **WHEN** ユーザーが本文翻訳タブを初回表示または reopen する
- **THEN** UI は `translation_flow.translation` から翻訳モデル設定と user prompt を復元しなければならない
- **AND** 他 phase の設定値をそのまま流用してはならない

### Requirement: 本文翻訳タブは reopen 時に選択カテゴリと選択行を復元しなければならない
システムは、本文翻訳 phase の reopen 時に `選択カテゴリ` と `選択行` を復元しなければならない。tree 開閉状態やスクロール位置、ページ内の視覚位置は復元対象に含めてはならない。

#### Scenario: reopen 時に選択行まで復元される
- **WHEN** ユーザーが既存 task を reopen して本文翻訳タブを開く
- **THEN** UI は前回の `選択カテゴリ` と `選択行` を復元しなければならない
- **AND** tree 開閉状態とスクロール位置を復元してはならない

#### Scenario: system prompt は read-only で表示される
- **WHEN** ユーザーが選択中レコードの prompt を確認する
- **THEN** UI は system prompt 全文を表示しなければならない
- **AND** system prompt に編集入力や保存操作を与えてはならない
- **AND** user prompt だけを編集可能にしなければならない

### Requirement: 本文翻訳タブは `会話` `クエスト` `その他` の 3 区分で対象一覧を表示しなければならない
システムは、翻訳対象一覧を `会話` `クエスト` `その他` の 3 区分で表示しなければならない。`会話` と `クエスト` は tree 形式、`その他` は flat table 形式で描画しなければならない。

#### Scenario: 会話とクエストが tree で表示される
- **WHEN** ユーザーが `会話` または `クエスト` を開く
- **THEN** UI は親ノードと子行を持つ tree 一覧を表示しなければならない
- **AND** 子行選択に応じて右ペインの詳細を切り替えなければならない

#### Scenario: その他は flat table で表示される
- **WHEN** ユーザーが `その他` を開く
- **THEN** UI は tree 展開 UI を出してはならない
- **AND** `その他` はクエスト・会話以外の record を対象にしなければならない
- **AND** `recordType` `editorId` `sourcePlugin` を識別できる行一覧を表示しなければならない

### Requirement: 本文翻訳タブは選択行の原文、訳文、メタデータ、prompt を同時に確認できなければならない
システムは、一覧で選択されたレコードに応じて、右ペインに `原文` `訳文` `メタデータ` `Prompt` を同時に表示しなければならない。会話選択時は話者情報、クエスト選択時は quest ID や stage 情報を優先表示しなければならない。

#### Scenario: 選択行の詳細が右ペインへ表示される
- **WHEN** ユーザーが対象一覧から 1 行を選択する
- **THEN** UI は原文、現在の訳文、recordType、editorId、sourcePlugin を表示しなければならない
- **AND** 会話行では `speakerId / NPC 名`、クエスト行では `questId / stage index / objective` を追加表示しなければならない
- **AND** 参照コンテキスト panel は、会話行では `会話の属性` `NPC の属性` `ペルソナ` `参考単語` を表示し、クエスト行とその他行では `本文の属性` `参考単語` のみを表示しなければならない

#### Scenario: 翻訳完了後は訳文を手修正できる
- **WHEN** 翻訳実行が完了または partial failed で終了している
- **THEN** UI は訳文 textarea を再び編集可能にしなければならない
- **AND** 手修正結果を一覧の状態表示へ反映しなければならない

### Requirement: 本文翻訳タブは本文状態を `未翻訳` `AI翻訳済み` `確定` の 3 種で管理しなければならない
システムは、各本文の状態を `未翻訳` `AI翻訳済み` `確定` の 3 種で表示・管理しなければならない。翻訳成功行は `AI翻訳済み` とし、手修正内容は本文ごとの `確定` ボタン操作で保存したときだけ `確定` に遷移させなければならない。

#### Scenario: 本文ごとの確定操作でのみ保存される
- **WHEN** ユーザーが `AI翻訳済み` 行の訳文を編集する
- **THEN** UI は自動保存してはならない
- **AND** ユーザーが当該本文の `確定` ボタンを押したときだけ保存し、行状態を `確定` にしなければならない

#### Scenario: `確定` 行を再編集しても一覧ラベルは維持される
- **WHEN** ユーザーが `確定` 行の訳文を再編集する
- **THEN** UI は一覧ラベルを `確定` のまま維持しなければならない
- **AND** `取り消し` ボタンを表示しなければならない
- **AND** `取り消し` 押下時は未保存編集を破棄し、保存済みの確定本文を再表示しなければならない

### Requirement: 本文翻訳タブは translating 中に一覧、設定、phase 遷移をロックしなければならない
システムは、本文翻訳 phase が `translating` の間、一覧選択、カテゴリ切替、検索、フィルタ、モデル設定、prompt 編集、phase 遷移を無効化しなければならない。progress と `翻訳中` 状態を観測可能にしなければならない。

#### Scenario: translating 中は UI がロックされる
- **WHEN** ユーザーが `翻訳開始` を押して phase が `translating` になる
- **THEN** UI は左ペインの一覧操作と右ペインの訳文編集を無効化しなければならない
- **AND** `次へ` を無効化しなければならない
- **AND** progress 表示と `翻訳中` の状態 badge を表示しなければならない

### Requirement: 本文翻訳タブは未確定編集のまま対象変更や移動をさせてはならない
システムは、未確定の訳文編集がある状態で他行選択、カテゴリ切替、検索、ページ移動、phase 遷移が要求されたとき、警告モーダルを表示しなければならない。ユーザーが破棄を明示した場合のみ遷移を継続できる。

#### Scenario: 未確定編集があるときに警告を表示する
- **WHEN** ユーザーが未確定編集を残したまま他行選択または移動操作を行う
- **THEN** UI は `未確定の変更があります` を示す警告モーダルを表示しなければならない
- **AND** ユーザーが継続編集を選んだ場合は現在行に留まらなければならない

### Requirement: 本文翻訳タブは partial failed と empty を区別し、進行可否を明示しなければならない
システムは、partial failed、full failed、empty を別状態として表示しなければならない。partial failed では成功行の閲覧・手修正と failed 行だけの再実行を許可し、empty では no-op 完了として後続 phase へ進めなければならない。

#### Scenario: partial failed では再実行と次工程進行を両立する
- **WHEN** 翻訳結果の一部だけが失敗して phase が `partialFailed` になる
- **THEN** UI は failed 件数を summary と alert で表示しなければならない
- **AND** `failed のみ再実行` を有効にしなければならない
- **AND** 成功行の閲覧と手修正を継続できなければならない
- **AND** `次へ` を押したときに `未翻訳` が残る場合は警告モーダルを表示し、進行確認を要求しなければならない

#### Scenario: `次へ` 押下時は `未翻訳` だけを警告対象にする
- **WHEN** ユーザーが本文翻訳 phase で `次へ` を押す
- **THEN** UI は `未翻訳` 行が 1 件以上残る場合だけ警告モーダルを表示しなければならない
- **AND** 警告モーダルでは未翻訳件数のみを表示しなければならない
- **AND** `AI翻訳済み` と `確定` の行だけで構成される場合は警告モーダルを表示してはならない

#### Scenario: 対象 0 件では no-op 完了として進める
- **WHEN** 本文翻訳対象が 0 件である
- **THEN** UI は `表示できる翻訳対象はありません` を表示しなければならない
- **AND** `次へ` を有効にしなければならない
