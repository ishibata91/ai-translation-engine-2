# AITE2 Bug Fix

この skill は既存機能の不具合を再現し、`docs/` の正本と照合しながら、実装不備か仕様乖離かを切り分けて直すための skill。

## 使う場面
- 「〇〇を修正して」と既存機能の挙動修正を依頼された
- 「バグ: 〇〇が〇〇しない」と不具合報告を受けた
- 実装と `docs/` のどちらが正か切り分ける必要がある
- 回帰不具合、再現手順付き不具合、UI 上の観測差分を扱う

## 手順
1. 再現条件を固定する。
2. 期待挙動と実挙動を分けて記録する。
3. `docs/` から関連 spec を特定して読む。
4. UI を含む不具合で観測事実が足りない場合だけ、最小限のログを入れて `wails dev` と Playwright MCP で再現を固定する。
5. 仕様と実装を照合し、実装不備、仕様の古さ、仕様の曖昧さ、仕様同士の矛盾を切り分ける。
6. 仕様が曖昧または矛盾している場合は、勝手に決めずユーザー確認を挟む。
7. 仕様が古い、誤っている、または不足している場合は、仕様修正を先に同じ変更セットへ含める。
8. 挙動修正は影響範囲を絞って最小差分で行う。
9. 修正後は同じ再現手順で回帰確認する。
10. 対象レイヤーに応じた品質ゲートを通す。

## 参照資料
- 具体例は `references/examples.md` を読む。
- 記録テンプレートは `references/templates.md` を使う。
- 仕様乖離の見分け方は `references/spec-gap-checklist.md` を使う。

## 原則
- 作業は対話内でタスク化し、常に 1 ステップずつ進める
- 再現、仕様確認、原因切り分け、修正、回帰確認を一度に混ぜない
- 各ステップで観測事実、判断、次の 1 手を明確にする
- 実装だけを見て正解を決めない
- 先に再現と仕様確認を行い、修正はその後に行う
- 修正より先に失敗条件を明確にする
- UI を含む不具合では、原則として `wails dev` と Playwright MCP で観測事実を固定する
- UI を含む不具合で原因観測が不足する場合だけ、ログを最小限追加する
- 仕様が古い、誤っている、または不足している場合は、docs 修正とコード修正を同じ変更セットで扱う
- 仕様文書同士が矛盾する場合は、ユーザー確認なしに片方を採用しない
- 関連しないリファクタを混ぜない
- 必要な品質ゲートは省略しない
