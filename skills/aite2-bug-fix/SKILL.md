# AITE2 Bug Fix

この skill はバグ修正用。
今は簡易草案として、何を扱うかだけを定義する。

## 使う場面
- 既存機能が期待どおり動かない
- 回帰不具合を直したい
- エラーログ、再現手順、スクリーンショットがある
- 実装と仕様のどちらが正しいか曖昧

## 必須ルール
- `docs/` 配下の関連仕様を必ず確認する
- 実装だけを見て正解を決めない
- 仕様との乖離を見つけたら、コード修正だけで閉じない
- 仕様が古い、曖昧、または誤っている場合は、ユーザーに正しい仕様を確認する
- 仕様が古い、誤っている、または不足している場合は、仕様文書を修正し、その後に挙動も修正する
- UI を含む不具合では、観測用ログを仕込んだ上で `wails dev` を起動し、Playwright MCP で再現と確認を行う

## やること
- 期待挙動と実際の挙動を分けて書く
- 再現手順を固定する
- `docs/` から関連仕様を特定する
- 仕様と実装の乖離を検知する
- 実装が間違いか、仕様が古いか、仕様が曖昧かを切り分ける
- 仕様が曖昧または矛盾する場合はユーザーに正しい仕様を確認する
- 仕様が古い、誤っている、または不足している場合は仕様文書を先に修正する
- UI を含む不具合で、再現手順だけでは原因を観測できない場合は観測用ログを仕込む
- UI を含む不具合では `wails dev` でアプリを起動する
- UI を含む不具合では Playwright MCP で再現と確認を行う
- 影響範囲を絞る
- 原因候補を整理する
- 最小修正で直す
- 回帰確認を行う

## 基本フロー
1. 再現手順を固定する
2. 期待挙動と実際挙動を分ける
3. `docs/` 配下の関連仕様を読む
4. UI を含む不具合で、再現手順だけでは原因を観測できない場合は観測用ログを仕込む
5. UI を含む不具合では `wails dev` で対象画面を起動する
6. UI を含む不具合では Playwright MCP で再現し、観測事実を固定する
7. 仕様との乖離を検知する
8. 次のどれかを判断する
   - 実装が仕様から外れている
   - 仕様が古い
   - 仕様が曖昧で判断できない
9. 仕様が曖昧または誤っている場合は、ユーザーに正しい仕様を確認する
10. 仕様が古い、誤っている、または不足している場合は仕様を修正する
11. その仕様に合わせて挙動を修正する
12. UI を含む不具合では、同じ `wails dev` + Playwright MCP 手順で回帰確認する
13. 品質ゲートを行う

## 再現方法の標準
- まず再現条件を固定する
- UI を含む不具合で、再現手順だけでは原因を観測できない場合は最小限の観測ログを仕込む
- UI を含む不具合では `wails dev` で対象画面を起動する
- UI を含む不具合では Playwright MCP で画面操作と観測を行う
- UI を含む不具合では、修正後も同じ手順で再確認する

ユーザーへの手動確認依頼は、自動再現が難しい場合の例外とする。

## 仕様乖離の扱い
- 仕様どおりに動いていないなら、実装を修正する
- 現実の要件に対して仕様が古いなら、仕様を修正した上で実装も揃える
- 仕様文書同士が矛盾するなら、ユーザー確認なしに勝手に採用しない
- ユーザー報告だけを正として即修正しない。必ず `docs/` と照合する

## 原則
- 作業は対話内でタスク化し、常に 1 ステップずつ進める
- 再現、仕様確認、原因切り分け、修正、回帰確認を一度に混ぜない
- 各ステップで観測事実、判断、次の 1 手を明確にする
- 先に再現、後で修正
- 修正より先に失敗条件を明確にする
- 先に仕様を確認し、次に乖離を判断する
- UI を含む不具合で原因観測が不足する場合だけ、ログを最小限追加する
- UI を含む不具合では、再現と回帰確認を同じ `wails dev` + Playwright MCP 手順で揃える
- 関連しないリファクタを混ぜない
- 必要な品質ゲートは省略しない
- 仕様修正が必要なら、コード修正と同じ変更セットで扱う
